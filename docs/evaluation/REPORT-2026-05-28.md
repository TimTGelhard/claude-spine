# claude-spine stress-test report — 2026-05-28

Reporter: Claude Opus 4.7 (1M context), Claude Code session — auto mode, max effort, read-only audit.

## Executive summary

The product **delivers** on most of what it claims. Personalization is real (not decorative), security hooks actually enforce what 18f-security.md preaches, the routing-vs-loading architecture is plausibly token-efficient by construction, and the core 22 skills hold their router shape (32–108 lines each, average ~50). But three claims that the README and CLAUDE.md trade on don't survive contact with the actual code: **(1) "token efficiency is the central design constraint" is currently aspirational** — the LC1 benchmark harness has been run (`benchmarks/tokens/results/results.jsonl` + `REPORT.md` exist) yet `FIXES.md:13` still describes it as "gated on authorization" with no number quoted anywhere user-facing; **(2) the "Plans dir" profile field is decorative** — `spine-writeback.sh:64-67` hard-codes `docs/plans` + `docs/PROGRESS.md`, contradicting `19g-field-effects.md:45` which lists `spine-writeback.sh` as a consumer; **(3) Q4 push-back has 4 options that no chapter or skill actually branches on** — `19g:64` describes 3 phrasings ("Soft / Standard / Spar with me") that don't match the 4 Q4 labels, and **both** stack-flavor CLAUDE.md stubs hard-code "Spar with me" as rule #2 regardless of the user's pick. The biggest universality gap is the supposedly "neutral" default `settings.json`: it carries 4 `supabase` allowlist entries, 5 `vercel` entries, and `nextjs.org` / `supabase.com` / `vercel.com` WebFetch domains in the *default* install — the README admits it ("trim to your stack") but the bias is still the first thing a Java/Rust/iOS dev sees. The product is **NOT YET** ready for "operating-discipline layer for any Claude Code user" but is close — the top three fixes are LC1-publish-with-numbers, repair the three decorative-personalization fields, and ship a genuinely neutral default settings.json with stack-extras as the opt-in path.

## A. Big questions

### A1. Is token efficiency measurable, or aspirational marketing?

**Verdict:** Architecturally defensible, empirically **unproven in user-facing docs.**

**Evidence:** The four claimed mechanisms in `CLAUDE.md:23-35` all exist in code:
- Skills as routers: verified — all 22 `op-*` SKILL.md files run 32–108 lines (`op-visuals` 32, `op-onboard` 108, mean ~50), and the ones with depth route to adjacent files (`op-onboard` → `questions-essential.md` / `questions-deep.md` / `subscription-tune.md` / `hook-tune.md` / `extras-merge.md` / `handoff.md`, all loaded on-demand per `SKILL.md:18-27`).
- Default-off heavy plugins: verified — `global/settings.json:187-189` ships `frontend-design`, `playwright`, `vercel` set to `false`.
- Default-off bucket loop: verified three times — `profile-template.md:82` defaults `Bucket loop: off`, and `op-suggest/SKILL.md:11-17`, `op-curate-nudge/SKILL.md:14`, `op-bucket-router/SKILL.md:12` all gate on the `Bucket loop:` field with default-off fallback.
- Subscription-tuned `autoCompactWindow` + `effortLevel`: verified — `subscription-tune.md:9-19` ships a 9-row mapping table, and the SKILL flow is per-tier explicit-approval.

**But the proof is missing where it matters.** `FIXES.md:13` says "the actual spine-on vs spine-off run is gated on authorization (~$9–$15)" — yet `benchmarks/tokens/results/results.jsonl` exists, `benchmarks/tokens/REPORT.md` exists, and the `results/raw/` directory holds dozens of `*__on__*.json` and `*__off__*.json` files (with multiple `.err` failures, e.g. `pe-where-rule__on__r1.json.err`, `hk-typecheck-after-edit__off__r2.json.err`, `tc-bash-or-edit__off__r1.json.err`, `ap-future-helper__on__r1.json.err`). Either LC1 has been run and FIXES is stale, or the run is partial — neither story is what the public-facing README claims about "token efficiency as the central design constraint." A reader has no way today to verify the claim from this repo.

### A4. Are security rules enforced or merely documented?

**Verdict:** **Enforced — the catalog mostly survives contact with the hooks.**

**Evidence:**
- `block-env-staging.sh:21` uses a regex `(^|[[:space:]])(.*/)?\.env(\.[a-zA-Z0-9_.-]+)?($|[[:space:]])` that catches all the documented patterns (`.env`, `.env.local`, `foo/.env`, `-A .env.production`). `tests/hooks/test-block-env-staging.sh` exists (not run per the read-only mandate). The hook returns `permissionDecision: deny` via jq + the Claude Code hook protocol.
- `block-env-commit.sh` closes the gap by reading `git diff --cached --name-only` and matching `(^|/)\.env(\.[a-zA-Z0-9_.-]+)?$`. Notable: it has `set -uo pipefail` (no `-e`) so parse failures fall through silently — the documented "designed to NEVER block in error" stance (line 9-12). Defense-in-depth, not defense-in-paranoia.
- `global/settings.json:122-129` adds deny rules: `Read(./.env)`, `Read(./.env.*)`, `Read(**/.env)`, `Read(**/credentials.json)`, `Read(**/*_secret*)`, `Read(**/*token*)`, `Write(./.env)`, `Write(**/.env)`. This is the third layer.
- `18f-security.md`'s "always" rules are uniformly absolute (the file explicitly notes line 5: "most entries below are intentionally absolute"). The catalog is consistent with the hooks.
- **Edge case:** `Write(**)` is in the allow list (settings.json:84), which lets Claude write anywhere except `.env`. A creative prompt injection that writes to `~/.ssh/authorized_keys` or `~/.bashrc` would not be blocked by the deny patterns. Not a "bug" — it's the default-permissive trust model — but it deserves a note in INSTALL.md.
- **Extras-merge:** `extras-merge.md:81-93` writes a `jq` filter restricted to `.permissions.allow` and `.permissions.WebFetch` arrays with `unique`. The `SKILL.md:103-108` rule ("only permissions.allow + permissions.WebFetch arrays append-only") is **actually enforced by the merge code**, not just stated. ✓

### A5. Does the workflow discipline generalize beyond solo web-SaaS?

**Verdict:** **The discipline does; the scaffolding doesn't.**

**Evidence:** `chapters/workflow/06-feature-sizing.md:9-17` explicitly names six session types (Build / Debug / Refactor / Explore / Review / Explain) and admits in the table that the capacity table "Doesn't apply" to Explore and Explain. That's honest. But:

- `Session shape:` is captured in the profile (G1) but, as A3 noted, **no downstream skill reads it**. So a user who answers "Mostly debugging / brownfield" or "Mostly exploring / prototyping" gets the same Build-shaped scaffolding.
- For exploratory / one-off / research work, `op-spine-active/SKILL.md:21-22` correctly bails: "A user without any plan layout is probably doing exploratory or one-off work; ambient cold-start is for repeat plan-driven sessions." So the failure mode is silent (no harm), not aggressive (no false fire). That's the right default — but G1 is still a wasted question.
- **ML notebooks / embedded firmware / academic paper code:** No worked examples; `templates/examples/web-saas-next-supabase/` is the only stack-specific variant shipped. The agnostic skeletons in `templates/*.md` are usable.

### A7. Does the spine respect a Free-tier user?

**Verdict:** **Yes, with one caveat.**

**Evidence:** The Free path is the most carefully tuned of any tier.
- `subscription-tune.md:11` proposes `autoCompactWindow: 120000` (down from Pro's 180000) and `effortLevel: medium` (down from `high`) — Free users explicitly get *lower* defaults, not Pro hand-me-downs.
- `19f-subscription-aware.md` Free rows are consistently the most conservative: "Avoid parallel fan-out entirely" (line 45), "Strongly discourage `/loop`" (line 62), "Push earlier — every ~30% context use" (line 71), "Do not assume 1M access" (line 89).
- Bucket loop is default off (`profile-template.md:82`), so Free users never get unsolicited capture/curate machinery loaded.
- **The caveat:** the *neutral* default `settings.json` still has `effortLevel: "high"` and `autoCompactWindow: 180000` (settings.json:191-192). If the user runs the installer but skips `/onboard`, they get Pro-class defaults regardless of plan. The subscription-tune step is post-essentials (`SKILL.md:92`) — never runs if the user never starts `/onboard`. So a Free user who clones the repo, runs install.sh, and immediately starts working pays Pro tax until they happen to type `/onboard`. The `op-welcome` skill (`SKILL.md:27-35`) does prompt them, but it's advisory, not enforcing.

### A8. Is "for people already running real Claude Code sessions" honest?

**Verdict:** **Honest — but defines the audience by what the product *doesn't* serve, not by what it serves.**

**Evidence:** EXPLAINER.md goes the other way: written for "a guide for people who didn't grow up writing code" (line 2). The two documents target different audiences with no cross-reference. README's audience gate at line 5 ("For people already running real Claude Code sessions on real projects who want their workflow to stop leaking quality") plus the explicit "Not a tutorial" framing is honest enough — a true beginner would bounce off `INDEX.md`, `op-foundations`, and the seven shipped templates. The mismatch is that EXPLAINER's "grandma learning to code gets gentle, patient explanations" framing (line 141) is aspirational — the product doesn't actually deliver a grandma-grade interface. Two front doors, two audiences, no contract.

### A9. Single biggest broken promise?

**Verdict:** **`19g-field-effects.md:45` claims `spine-writeback.sh` reads the `Plans dir:` field; the script (`global/hooks/spine-writeback.sh:64-65`) hard-codes `docs/plans` and `docs/PROGRESS.md`.**

This is the single sharpest gap-between-claim-and-reality I found. The promise: "If you set `Plans dir: project/x/plans`, ambient cold-start and writeback both honor that path." (19g:45). The reality: line 64 of the hook is `[ -d "$CWD/docs/plans" ] || exit 0`. A user trusting the docs would set `Plans dir: roadmap/` in their profile, get zero behavior change, and have no way to learn this is decorative short of reading the hook source. The `Plans dir` row in `19g` should either be removed (option B) or the hook should grow a profile-reading loop matching what `spine_default_int` does for thresholds (option A — verified at lines 45-57 of the hook, where threshold reading IS wired).

### A10. Single biggest unstated assumption?

**Verdict:** **"You operate `bash + jq + git` on macOS/Linux comfortably enough to debug `~/.claude/settings.json` by hand."**

The whole personalization layer assumes the user will read three hook-merge fallback messages (`extras-merge.md:99-117`, `hook-tune.md:128-148`, plus the Hand-edit fallback paths) calmly and edit JSON by hand. The "Plain English" framing of EXPLAINER.md doesn't survive contact with what happens when `extras-merge.md`'s jq merge fails. A reader matching EXPLAINER's "grandma learning to code" target will see "Couldn't auto-merge `+vcs-gitlab.json`... append the following entry to the existing matcher's `hooks` array, then restart Claude Code" and be lost.

## J. What's missing — top 3 gaps

1. **A worked `templates/examples/<library-or-cli>` variant.** The product claims universality but ships one worked example (web SaaS Next/Supabase). The agnostic skeletons are real, but a CLI or library publish DEPLOY.md variant would close the most obvious "this is for web devs" objection in 30 minutes. A `/profile verify` command that reads each field and reports "consumer present" / "consumer missing" would close this — or, less ambitiously, a "tested-in-CI" annotation in `19g`'s table for fields that have a test fixture.

3. **A `--free` install variant.** A Free user who skips `/onboard` gets Pro-class defaults. A shipped Free-class settings.json (lowered `effortLevel`, lowered `autoCompactWindow`) would close the silent-Pro-tax path. Easier than guessing what tier the user is.

## Verdict

1. **Is the project ready for the public-launch claim?** **NOT YET.** The architecture is solid, the security stance survives scrutiny, the personalization machinery is real where it claims to be, and the discipline-applied-to-self mostly holds. But three loud lies remain: (1) the "central design constraint" of token efficiency has no published number despite the harness having results on disk, (2) three personalization fields are decorative despite `19g` claiming named consumers, and (3) the "neutral by default" install ships ~10 stack-specific allowlist entries that betray the "any stack, any platform" claim. These are all 1–3 hour fixes, but each is the kind of thing a careful reader of the README would catch in the first hour after install.

2. **Top 3 highest-impact fixes before LC5:**
   - **LC1a (new):** Publish the LC1 benchmark numbers in `benchmarks/tokens/REPORT.md` and reference them from the README ("X% input-token reduction vs spine-off across N prompts"). Or kill the run and re-frame the headline as architectural-only. The current state (results on disk, FIXES still listing it open) is the worst of both worlds.
   - **BA4 (new):** Repair the three decorative profile fields. Either wire Plans dir into `spine-writeback.sh` (15 min, mirrors `spine_default_int`), wire Q4 push-back into `op-signaling` and the stack-flavor stubs, and wire Session shape (G1) into `op-collaboration-modes`; or strike the rows from `19g`. The "I changed X and nothing happened" reaction is precisely the failure 19g exists to prevent.
   - **BA5 (new):** Ship a genuinely neutral default `global/settings.json` — strip the 4 supabase / 5 vercel / nextjs+supabase+vercel WebFetch entries and move them into a `+web-saas-stack.json` settings-extra fragment. The current state contradicts the README's neutrality claim on first read.

3. **Single piece of evidence that would change the verdict:** **A run of the LC1 benchmark with published numbers showing >20% input-token reduction across the eval set, plus a sweep of `19g` against actual code-grep evidence (or removal of the three decorative rows).** Both are cheap. If those two land, the product becomes "operating-discipline layer for any Claude Code user" in fact, not just in framing.
