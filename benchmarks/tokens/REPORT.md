# Token-efficiency benchmark — REPORT

**LC1 run — Sonnet 4.6, 2026-05-28.** Spine-on vs spine-off, `claude -p` single-shot.
n = **11 paired prompts** (30 spine-off + 33 spine-on calls). Total spend ≈ **$6.48** / 87 successful calls.

> **Curated publication.** The per-prompt / cache / variance tables below are
> `aggregate.py` output over the published pairs (`results/results.jsonl`, filtered
> to the 11 prompts with surviving pairs). The **headline, per-load worst-case, and
> caveats are curated** and are *not* regenerated by `aggregate.py` — a bare
> `python3 aggregate.py` re-run overwrites this file with tables only (and its
> sum-based **Totals** row is not the headline — see [Why not the Totals row](#why-not-the-totals-row)).
> Full provenance incl. dropped calls: `results/archived-pre-publish/results.full.jsonl`.

## Headline

**The spine does *not* use fewer tokens than vanilla Claude Code. It uses more.**

On the 11 paired prompts (call-weighted means, Sonnet 4.6):

| metric | spine-off | spine-on | Δ | Δ % |
|---|---:|---:|---:|---:|
| input tokens / call | 58,979 | 104,148 | +45,169 | **+76.6 %** |
| output tokens / call | 1,015 | 696 | −319 | **−31.5 %** |
| cost (USD) / call | $0.0680 | $0.0875 | +$0.0195 | **+28.6 %** |

So the naive framing — *"the spine saves tokens by loading on demand"* — is **refuted**
by measurement. Loading the spine costs **+77 % more input tokens and +29 % more dollars
per call** than running with no spine at all.

**What the spine actually buys, and what holds up under measurement:**

1. **Bounded per-load cost.** The heaviest single on-demand load (`op-onboard --deep`,
   eight files) is ≈ 8.2k tokens at the 7-tokens/line proxy — **4.1 % of Sonnet 4.6's
   200k window** (≈ 2.9 % at the LC1-measured ~5 tokens/line). Every top-5 worst-case
   stack sits in `chapters/02-context-budget.md`'s green zone. See
   [Per-load worst-case](#per-load-worst-case-the-constraint-that-holds).
2. **Routing accuracy** — whether the right skill fires on the right query — is the
   spine's real value-add, and it is **not measured here**. See
   [`tests/skill-triggers/`](../../tests/skill-triggers/).
3. **More focused answers.** Spine-on output was **−31.5 %** (1,015 → 696 tokens/call):
   guidance produced tighter responses. On the one brownfield prompt
   (`bf-inherited-project`), spine-on input was actually **−21.7 %** — with guidance,
   Claude stopped blind-exploring the codebase.

The honest claim is **bounded per-load cost + routing accuracy**, not *"fewer tokens
than vanilla."* The +45k/call delta is the spine's always-on session footprint (the
23-skill routing manifest loaded every session for classification, the global
`CLAUDE.md`, and any ambient skill that fires on session start) — not the on-demand
chapter loads, which F3 bounds at ≤ 8.2k. The benchmark measures *total* per-call input;
F3 isolates only the on-demand slice. The two answer different questions; don't conflate them.

### Reconciliation with audit-02 F0

This report reproduces audit-02 `docs/plans/audit-02-token-cost.md` § Findings **F0** to the decimal:

- **Input +75.9 %** (F0's headline) = the **per-prompt mean-of-means** (each of the 11
  prompts weighted equally): off 59,209 → on 104,148. Disk match: 59,209.48 → 104,147.88.
- **Input +76.6 %** (this report's headline) = the **call-weighted** mean (each of the 30/33
  calls weighted equally): off 58,979 → on 104,148. The 0.7 pp gap is purely the OFF
  baseline aggregation — `ap-future-helper` (1 surviving OFF run) and `bf-inherited-project`
  (2) are under-weighted in the call-weighted mean.
- **Cost +28.6 %** and the cache picture (off cc 9.6k / cr 49k · on cc 12.6k / cr 91.5k)
  match F0 exactly.

### Why not the Totals row

`aggregate.py`'s **Totals** section sums tokens across calls, so its Δ % is sensitive to
unequal call counts (33 spine-on calls vs 30 spine-off — `ap-future-helper`/`bf-inherited-project`
have < 3 surviving OFF runs). Its sum-based input Δ is **+94.2 %**, *not* the +76.6 % per-call
figure. The per-call mean above is the apples-to-apples number; the Totals row is retained
below only as raw evidence.

## Per-load worst-case (the constraint that holds)

From audit-02 **F3** — combined router + routed chapters + adjacent files + templates,
line counts then proxy tokens @ 7 tokens/line:

| rank | load | lines | ≈ tokens | % of 200k |
|---|---|---:|---:|---:|
| 1 | `op-onboard --deep` (8 files) | 1172 | 8.2k | **4.1 %** |
| 2 | `op-prepare` (SKILL + procedure + 05h/i/j + 5 templates) | 986 | 6.9k | 3.5 % |
| 3 | `op-onboard` essentials (5 files) | 576 | 4.0k | 2.0 % |
| 4 | `op-foundations` + 3 default chapters + topic | 235 | 1.6k | 0.8 % |
| 5 | `op-anti-patterns` + 18-meta + topic | 131–172 | 1.0–1.2k | 0.5–0.6 % |

All five within the green zone. The LC1 cache-read delta gave a **measured** ratio of
~5 tokens/line (vs the 7 proxy), so these figures are ~30 % high — the heaviest stack is
nearer **5.9k tokens (2.9 %)** in practice. The claim *"loading any single spine path stays
a small fraction of the context window"* holds; the claim *"the spine uses fewer tokens
overall"* does not.

## Dropped runs

**87 of 171 logged calls succeeded; 84 were dropped.** The failure was external, not a
harness bug (audit-02 F5):

- A mid-run `claude` CLI auto-upgrade (2.1.153 → 2.1.154, ~21:13 on 2026-05-28) replaced the
  binary mid-execution. The spine-off batch had completed 30 of 57 calls; the remaining
  **27 spine-off + all 57 spine-on** calls hit `exit 127` (`No such file or directory`).
- The spine-on batch was recovered via a `--only-cond on` re-run (57 calls, all ok). The
  spine-off batch was **not** re-run, so 8 prompts kept only spine-on data.
- **Net: 30 spine-off ok + 57 spine-on ok = 87.** Of these, **11 prompts have surviving
  pairs** (30 off + 33 on → the headline). **8 prompts are spine-on-only** (their off runs
  were among the dropped 27) and are excluded from every delta: the 4 negative controls
  (`ctrl-*`) plus `hk-typecheck-after-edit`, `pr-bad-output`, `sa-parallel-subagents`,
  `sg-scope-check`.

The dropped/ON-only raw JSON and the full 171-record log live under
`results/archived-pre-publish/`. `run.sh` now aborts on a mid-run binary change (preflight
version + mtime guard) so this exact failure can't silently corrupt a future run.

## What this doesn't measure

- **Routing accuracy is the spine's actual job — and it's measured elsewhere.** This
  benchmark measures input-context *cost*, not whether skills fire on the right queries.
  A spine that loads with bounded cost *and* routes accurately is doing its job; for the
  routing half, see [`tests/skill-triggers/README.md`](../../tests/skill-triggers/README.md).

- **The negative-control signal was lost.** The `ctrl-*` prompts exist specifically to test
  for spine overhead on prompts the spine *can't* help (pure-code questions). Their spine-off
  runs were among the dropped 27, so the controls are spine-on-only here — the
  "overhead-on-irrelevant-prompts" delta is **unmeasurable in this LC1 run**. (Spine-on,
  the controls loaded ~33k vs ~103k for spine-relevant prompts, i.e. the spine loaded far
  less on prompts it couldn't help — but with no off baseline that's suggestive, not a delta.)

- **Multi-turn compounding is not captured.** A single `claude -p` reloads the full context
  every time. In an interactive session the spine's per-turn footprint amortizes across many
  turns against a warm cache; real per-turn cost is lower than these single-shot numbers.

- **The spine state was 22 of 23 op-* skills.** `op-approach` (built Section 0, 2026-05-28)
  was repo-only, not in the installed spine at run time. ~0.5 % of the spine-on input;
  negligible against the +45k delta, but the run reflects the pre-`op-approach` spine.

- **`aggregate.py`'s Totals row is sum-based.** Over unequal/unpaired call counts it
  over-reports the delta (+94.2 % vs the +76.6 % per-call mean). Use the per-call Headline
  table, not the Totals row, as the comparison.

- **Cost is the proxy, tokens are the truth.** Sonnet 4.6 / Opus / Haiku weight input vs
  output vs cache tokens differently. Read Δ tokens and Δ cost together — if they diverge,
  the model assumption changed.
